# 客服消息推送架构同步

日期：2026-05-01
维护方：Codex
阅读对象：后端 Claude

## 核心结论

当前客服方案不能继续理解为“纯自研小程序聊天页”。

如果只使用我们自己的 `pages/contact/index/index` + 自建 API：

- 用户在线时可以聊天。
- 用户退出小程序后，前端连接不可依赖。
- 后端只能保存未读消息，等待用户下次打开小程序拉取。
- 不能保证在抖音主界面实时触达用户。

要实现用户退出小程序后仍能通过抖音消息中心复返，必须接入抖音官方的「消息推送客服」通道。

推荐架构：

```text
自研客服系统 + 抖音消息推送客服官方通道
```

## 后端需要立即确认的问题

1. 当前小程序后台是否已经可以配置「消息推送客服」。
2. 消息推送客服是否支持当前小程序类目。
3. 回调消息 payload 最终字段是什么。
4. 发送客服消息接口当前支持哪些消息类型。
5. 48 小时 / 5 条限制的错误码和失败返回结构。
6. 是否支持用户在抖音「消息」-「第三方客服消息」复返到会话。
7. 是否允许把消息转发到我们现有客服系统并由我们系统坐席回复。

## 对前端的影响

前端后续应增加客服模式配置：

```ts
export const CUSTOMER_SERVICE_MODE = 'douyin_message_push';
```

可选模式：

```ts
'douyin_message_push' | 'douyin_im' | 'self_hosted'
```

其中：

- `douyin_message_push`：目标正式方案。
- `douyin_im`：审核最稳的官方 IM 兜底方案。
- `self_hosted`：仅作为开发 mock 或官方能力不可用时的兜底，不承诺退出后推送。

## 当前行动建议

后端 Claude 应优先基于 `docs/customer-service-message-push-architecture.md` 调整客服后端设计，而不是继续按纯 WebSocket/自建 IM 方案设计完整离线推送能力。
